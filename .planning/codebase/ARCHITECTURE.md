# Planning Architecture for adapterOS

This reference point summarizes the live topology, control/worker boundaries, and where to plant future code. Use the existing `docs/ARCHITECTURE.md` as the canonical deep dive, but rely on the paths below when assigning work.

## Control plane boot & HTTP surface
- Boot phases live in `crates/adapteros-server/src/main.rs` and the `adapteros_server::boot` modules under `crates/adapteros-server/src/boot/`. `StartupOrchestrator::run_phase` there wires the config → security → executor → invariants → database → lifecycle flow, so any new invariant, preflight check, or boot hook belongs next to `run_preflight_checks`, `validate_boot_invariants`, or the specific `boot/*.rs` file that owns the phase logic.
- Configuration and secrets drive `crates/adapteros-config` plus the `configs/` tree (e.g., `configs/cp.toml`, `configs/dev.toml`, `configs/production-multinode.toml`). New configuration knobs should be defined in `crates/adapteros-config/src/lib.rs:path_resolver.rs` and wired into `adapteros-server::boot::initialize_config` so the CLI (`crates/adapteros-server/src/cli.rs`) can expose them.
- The HTTP surface anchors on `crates/adapteros-server-api`. Routes live under `crates/adapteros-server-api/src/routes/` (flat files such as `handlers.rs`, `health.rs`, `routes/optional.rs`), middleware under `crates/adapteros-server-api/src/middleware/` (see `chain_builder.rs` for the enforced type-state order), and inference wiring under `crates/adapteros-server-api/src/inference_core/core.rs`. When you add a new endpoint or policy-sensitive hook, extend the appropriate file in `routes/` and register the middleware in `middleware/chain_builder.rs` before touching the handler.
- OpenAPI generation is triggered from `crates/adapteros-server/src/main.rs` via `src/openapi.rs` and lands in `openapi.json`. Keep new public routes mirrored in `openapi.rs` so the CLI and UI clients (`crates/adapteros-cli`, `crates/adapteros-ui`) pick up the spec automatically.

## Data plane + inference backends
- Worker entry points live in `crates/adapteros-lora-worker/src/lib.rs` and `src/uds_server.rs`; the Unix-domain socket transport path is defined via `crates/adapteros-core/src/defaults.rs` (see `DEFAULT_WORKER_SOCKET_PROD_ROOT`). Any new worker RPC belongs next to `WorkerInferRequest/Response` inside `crates/adapteros-lora-worker/src/uds_server.rs` or supporting modules.
- Adapter selection, scoring, and quantization happen in `crates/adapteros-lora-router` (look at `src/lib.rs` and `quantization.rs`). When adding new router features, keep the score/seed logic in this crate and trigger it from `adapteros-server-api::inference_core` so the request path still goes `InferenceCore → Router → UDS`.
- Backend kernels are split across `crates/adapteros-lora-mlx-ffi` (primary MLX glue), `crates/adapteros-lora-kernel-mtl`, and `crates/adapteros-lora-kernel-coreml`. New backend code should live in the crate that matches the target GPU/runtime; the worker dispatch logic in `crates/adapteros-lora-worker/src/lib.rs` already enumerates which backend to call, so extend the dispatcher there and add the runtime-specific code into the matching kernel crate.
- Adapter lifecycle, hot-swap, and registry hooks surface in `crates/adapteros-lora-lifecycle` and `crates/adapteros-registry`. When adding storage or eviction rules, add tests under `crates/adapteros-lora-lifecycle/src/tests` and keep registry migrations under `migrations/` so the control plane can replay them.

## Storage, telemetry, and policies
- Runtime state lives inside `var/`: sockets in `var/run/aos`, adapters under `var/adapters/`, SQLite registries such as `var/aos-cp.sqlite3`, and telemetry buckets in `var/telemetry/`. This directory is git-ignored, so mutate it through the boot code in `crates/adapteros-server` or the worker crates, not by checking in files directly.
- Database logic and migrations for the control plane rely on `crates/adapteros-db` and the `migrations/` folder (see `migrations/2025_*.sql` for patterns). Schema or telemetry changes must accompany a new migration file alongside an update to `adapteros-server::boot::run_migrations`.
- Policies load from `crates/adapteros-policy/src/registry.rs` (see `PolicyId` enumerations) and are enforced by `crates/adapteros-server-api/src/middleware/policy_enforcement.rs`. Anything changing how a policy is evaluated should add a new entry to the registry and the middleware configuration there.
- Telemetry and audit supporters live under `crates/adapteros-telemetry`, `crates/adapteros-telemetry-types`, and `crates/adapteros-telemetry-verifier`. If you add new metrics or evidence hooks, define the metric/event type in `adapteros-telemetry-types` and extend `adapteros-telemetry/src/lib.rs` plus the `telemetry_ext.rs` helpers in `crates/adapteros-server-api/src/telemetry/`.
