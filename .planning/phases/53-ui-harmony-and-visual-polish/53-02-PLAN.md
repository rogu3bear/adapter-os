  <action>
    **Audit methodology (per RESEARCH.md Pattern 1):**

    For EACH surface (chat workspace, then dashboard), read every Rust component file and enumerate:
    1. Every visible element (buttons, labels, sections, cards, badges, dividers, icons)
    2. Classify each element: **essential** (connects to working feature, actively serves operator) / **redundant** (duplicate info shown elsewhere) / **dead** (onClick does nothing, links to unimplemented feature, placeholder) / **orphaned** (references removed feature, shows stale data)
    3. Check glass tier assignment: is each element at the correct tier? (Tier 1 for cards/content, Tier 2 for panels/sidebars, Tier 3 for modals/overlays)
    4. Check spacing: does it use design tokens (`var(--space-*)`) or hardcoded px?
    5. Check typography: correct font weight, size, line-height for element role?
    6. Check interaction states: does every clickable/focusable element have hover/active/focus-visible?
    7. Check skeleton loader: does it match content layout?
    8. Check empty state: does it use `<EmptyState>` with centered text + hint?
    9. Apply squint test: is visual hierarchy clear?

    **Before classifying any element as dead/orphaned:**
    - Grep for its `data-testid` or ID across `tests/`, `crates/adapteros-e2e/`, and the codebase
    - Check `crates/adapteros-ui/src/components/command_palette.rs` and `src/components/search/` for route-based action references
    - Check if any signal/callback wires to it from parent components

    **Output:** Create a structured audit report listing every element found, its classification, and proposed action (keep/remove/fix). Format as a markdown table in the checkpoint message. Group by surface (Chat, Dashboard).

    **Do NOT make any code changes in this task.** This is audit-only. Changes happen in Task 3 after user approval.
  </action>

  <action>
    **Implement the user-approved cut list from the checkpoint:**

    For each approved removal:
    - Remove the Rust view element entirely (not just `display: none`)
    - If the element owned a signal/callback that nothing else uses, remove that too
    - If removing the element changes the layout flow, adjust the parent container's spacing
    - Use `<Show>` not `{move || if ...}` for any conditional rendering changes (per CLAUDE.md signal disposal rules)
    - Use `.try_get().unwrap_or_default()` in any new or modified reactive closures

    For each approved fix:
    - Fix the specific issue (wrong glass tier, missing hover state, hardcoded spacing, etc.)
    - Apply the fix using existing design tokens and component library patterns

    **Glass tier enforcement:**
    - Session list sidebar panel: Tier 2 (`glass-bg-2`) — it's a panel/sidebar
    - Chat conversation area: Tier 1 (`glass-bg-1`) — it's content
    - Composer input area: Tier 1 — it's content
    - Any modal/overlay within chat: Tier 3 (`glass-bg-3`)
    - Dashboard cards: Tier 1 — they're content
    - Dashboard stats/summary section: Tier 1

    **Interaction state audit and fix:**
    - Every button must have hover (translateY(-1px) + shadow), active (translateY(1px)), and focus-visible (2px ring) states. Use existing `.btn` CSS patterns from core.css.
    - Every clickable card must have hover state (subtle background change or elevation shift)
    - Session list items must have hover and selected states
    - Composer send button must have clear enabled/disabled visual distinction

    **Spacing consistency:**
    - Replace any remaining hardcoded `px` padding/margin with `var(--space-*)` tokens
    - Ensure consistent gap between chat elements

    **Rebuild CSS bundle:** Run `bash scripts/bundle-css.sh` after any CSS changes.

    **Critical safety checks:**
    - Do NOT remove elements with `data-testid` attributes unless the audit confirmed no test references them
    - Do NOT change any signal types or API call shapes
    - Do NOT modify the SSE event handling or streaming inference logic
    - Preserve all `aria-label`, `role`, and accessibility attributes. If removing a dead button that had an aria-label, that's fine — the label goes with the element.
