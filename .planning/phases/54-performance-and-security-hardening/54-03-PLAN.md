  <files>

  <action>
    1. Define eviction event in `crates/adapteros-transport-types/src/lib.rs`:
       - Add an `AdapterEvicted` variant to the SSE event enum (or create one if transport-types doesn't have SSE events)
       - Fields: `adapter_id: String`, `adapter_name: Option<String>`, `reason: String` (e.g., "memory_pressure"), `timestamp: String`
       - Use `#[serde(default)]` on all new fields for backward compatibility per SSE event contract in CLAUDE.md

    2. Emit eviction event from `MemoryPressureManager`:
       - In `crates/adapteros-memory/src/pressure_manager.rs`, when `evict_low_priority` or equivalent eviction method succeeds:
       - Check if `eviction_notifications` config flag is true (from MemoryConfig added in Plan 01)
       - If true, send the eviction info through a channel (add a `tokio::sync::broadcast::Sender<EvictionEvent>` to `MemoryPressureManager`)
       - The channel approach decouples memory management from SSE transport — the API server subscribes to the channel and forwards to SSE clients

    3. Forward eviction events to SSE in `crates/adapteros-server-api/src/handlers/streaming_infer.rs`:
       - Add the `AdapterEvicted` event to the `InferenceEvent` enum (which is the SSE event type used in streaming)
       - When the API server's eviction channel receiver gets an event, inject it into active SSE streams
       - If injecting into active streams is complex, alternatively emit it as a system-level SSE endpoint (e.g., `/v1/events/system`) that the UI polls or subscribes to

    4. Handle eviction event in UI:
       - In `crates/adapteros-ui/src/signals/chat.rs`, add a handler for `AdapterEvicted` events from the SSE stream
       - Use `#[serde(default)]` on the new event variant for backward compat
       - When received, set an eviction signal that triggers a toast

    5. Create or extend toast component in `crates/adapteros-ui/src/components/toast.rs`:
       - If toast component exists, add eviction toast variant
       - If not, create a minimal toast component:
         - Displays "Adapter {name} unloaded to free memory" (brief, non-blocking per user decision)
         - Auto-dismisses after 5 seconds
         - Uses Liquid Glass design system (no Tailwind — pure CSS)
         - Uses `<Show>` for conditional rendering (not `{move || if ...}`)
         - Uses `try_get()` / `try_set()` for all signal access (per Leptos 0.7 rules in CLAUDE.md)
         - Uses `gloo_timers::callback::Timeout` for auto-dismiss timer (not `std::time`)

    Avoid: Do NOT make eviction toast blocking (no confirmation required). Do NOT use `println!` in UI code (use `leptos::logging::log!`). Do NOT use `.get()` / `.set()` on signals (use `try_get()` / `try_set()` — signals dispose on component unmount).
  </action>

    <manual>Verify AdapterEvicted event type is defined and toast component renders</manual>
  </verify>
  <done>

    - MemoryPressureManager emits eviction events via broadcast channel when config enables it

    - Toast follows Liquid Glass design system, uses correct Leptos 0.7 patterns
  </done>
