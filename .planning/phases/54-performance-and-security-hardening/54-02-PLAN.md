  <action>
    1. Extend `RateLimitsConfig` in `crates/adapteros-config/src/types.rs` with per-tier RPM fields:
       ```rust
       pub struct RateLimitsConfig {
           pub requests_per_minute: u32,       // existing -- default for unspecified tiers
           pub burst_size: u32,                // existing
           pub inference_per_minute: u32,      // existing
           #[serde(default)]
           pub health_rpm: Option<u32>,        // None = unlimited (health probes bypass middleware already)
           #[serde(default)]
           pub public_rpm: Option<u32>,        // e.g., 600 for login/status
           #[serde(default)]
           pub internal_rpm: Option<u32>,      // e.g., 1000 for worker heartbeats
           #[serde(default)]
           pub protected_rpm: Option<u32>,     // e.g., 300 for all protected routes
       }
       ```
       All new fields are `Option<u32>` with `#[serde(default)]` for backward compatibility -- existing cp.toml files without these fields will deserialize to None, falling back to `requests_per_minute`.

    2. Update `configs/cp.toml` `[rate_limits]` section with commented per-tier defaults:
       ```toml
       [rate_limits]
       requests_per_minute = 300
       burst_size = 60
       inference_per_minute = 150
       # Per-tier overrides (omit to use requests_per_minute as default)
       # health_rpm =          # health routes bypass rate limiting
       public_rpm = 600        # /v1/auth/login, /v1/status, /metrics
       internal_rpm = 1000     # /v1/workers/register, /v1/workers/heartbeat
       protected_rpm = 300     # all protected routes (full middleware chain)
       ```

    3. Modify `rate_limiting_middleware` in `crates/adapteros-server-api/src/middleware_security.rs`:
       - Add a helper function `fn route_tier(path: &str) -> RouteTier` that classifies a request path into one of: Health, Public, Internal, Protected
       - Health: paths starting with `/healthz`, `/readyz`, `/version` (these already bypass middleware, but document)
       - Public: `/v1/auth/`, `/v1/status`, `/metrics`
       - Internal: `/v1/workers/`
       - Protected: everything else
       - In the rate limit check, resolve the tier-specific RPM: `config.public_rpm.unwrap_or(config.requests_per_minute)` etc.
       - Pass the resolved RPM to `check_rate_limit` (modify the function signature if needed to accept a per-request limit, or compute the limit before calling)

    4. Review and tighten `RATE_LIMIT_EXEMPT_PATHS`:
       - Per research pitfall 3: entries like `/v1/system/` and `/v1/models/` may exempt too many routes
       - Only GET status/health routes should be exempt
       - Model mutation routes (POST/DELETE) should NOT be exempt
       - If the exempt list is overly broad, narrow it to exact paths or add method checks

    5. Implement custom `Debug` for `SecurityConfig` that redacts `jwt_secret`:
       - Remove `#[derive(Debug)]` from `SecurityConfig`
       - Add `impl fmt::Debug for SecurityConfig` that prints all fields except `jwt_secret`, which should show `"[REDACTED]"`
       - This prevents accidental secret leakage via debug logging (research pitfall 4)
  </action>

       - `#!/usr/bin/env bash`, `set -euo pipefail`
       - Header banner: "Security Audit Contract Check"

       Checks (HARD FAILURES -- exit 1):
       a. Auth enforcement: Verify all protected route handlers reference auth middleware (grep for auth extractor in handler files). Flag any protected handler that doesn't extract auth claims.
       b. Dev bypass production guard: Verify `dev_bypass_status()` is checked and blocks in release builds (already in `check_release_security_assertions.sh`, but duplicate the critical check here for completeness).
       c. Rate limiting coverage: Verify `rate_limiting_middleware` is in the global middleware chain (grep `routes/mod.rs` for the middleware layer).
       d. Input validation: Check that POST/PUT/PATCH handlers use typed request bodies (not raw `String` or `Json<serde_json::Value>`) by scanning handler signatures.
       e. CSRF protection: Verify CSRF middleware is applied to protected routes.

       Checks (WARNINGS -- print but don't fail):
       f. Missing `#[validate]` annotations on request body structs (informational).
       g. Routes with overly broad rate limit exemptions.

       Output: Summary table of checks (PASS/FAIL/WARN), exit 1 if any hard failure.

    2. Create `scripts/contracts/check_secret_exposure.sh`:
       - `#!/usr/bin/env bash`, `set -euo pipefail`
       - Header banner: "Secret Exposure Scanner"

       Checks (HARD FAILURES):
       a. Grep source files (excluding target/, var/, .git/) for patterns that suggest hardcoded secrets:
          - `password\s*=\s*"[^"]+"` (hardcoded passwords)
          - `secret\s*=\s*"[^"]+"` (hardcoded secrets -- exclude TOML config patterns like `jwt_secret = ""`)
          - `Bearer [A-Za-z0-9._-]+` in source (not test fixtures)
          - `api_key\s*=\s*"[^"]+"` (hardcoded API keys)
       b. Check that `SecurityConfig` does NOT derive `Debug` (should have custom impl from Task 1):
          - Grep for `#[derive(` on SecurityConfig and verify `Debug` is not in the list
       c. Check for `tracing::debug!` or `tracing::info!` calls that reference config structs with sensitive fields:
          - Pattern: `debug!.*config` or `info!.*security_config` in server code
       d. Verify `var/models/` has appropriate .gitignore (model weights should never be committed).

       Checks (WARNINGS):
       e. Any `Debug` derive on structs with fields named `secret`, `key`, `token`, `password`.
       f. Log grep for common secret patterns in test output (if var/logs/ exists).

       Output: Summary table, exit 1 if any hard failure.
