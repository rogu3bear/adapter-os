  <action>
    1. Extend `RateLimitsConfig` in `crates/adapteros-config/src/types.rs` with per-tier override fields:
       ```rust
       pub struct RateLimitsConfig {
           pub requests_per_minute: u32,    // existing — becomes default/protected tier
           pub burst_size: u32,             // existing
           pub inference_per_minute: u32,   // existing
           // New per-tier overrides (None = use default `requests_per_minute`)
           #[serde(default)]
           pub health_rpm: Option<u32>,     // None = unlimited (health probes)
           #[serde(default)]
           pub public_rpm: Option<u32>,     // e.g., 600 for login/status
           #[serde(default)]
           pub internal_rpm: Option<u32>,   // e.g., 1000 for worker heartbeats
           #[serde(default)]
           pub protected_rpm: Option<u32>,  // e.g., 300 for protected routes
       }
       ```
       Keep backward compatibility — existing configs without new fields work via `#[serde(default)]`.

    2. Update `RateLimiterConfig` in `crates/adapteros-server-api/src/rate_limit.rs`:
       - Add a `for_tier(tier: RouteTier, config: &RateLimitsConfig) -> Self` constructor that selects the appropriate rpm based on tier
       - `RouteTier::Health` returns a config with very high or effectively unlimited rpm (e.g., u32::MAX)
       - `RouteTier::Public` uses `config.public_rpm.unwrap_or(config.requests_per_minute)`
       - `RouteTier::Internal` uses `config.internal_rpm.unwrap_or(config.requests_per_minute)`
       - `RouteTier::Protected` uses `config.protected_rpm.unwrap_or(config.requests_per_minute)`
       - If `RouteTier` enum doesn't exist, create it as a simple enum in the rate_limit module

    3. Wire per-tier configs in route builder (`crates/adapteros-server-api/src/routes/mod.rs` or wherever the Axum router is constructed):
       - When constructing rate limiting middleware for each route group, pass the appropriate tier
       - Health routes: skip rate limiting or use unlimited config
       - Public routes: use public_rpm
       - Protected routes: use protected_rpm
       - Check existing `RATE_LIMIT_EXEMPT_PATHS` — ensure health paths remain exempt

    4. Verify fail-closed auth enforcement:
       - In the auth middleware, confirm that when `AOS_DEV_NO_AUTH` is NOT set and auth fails, the response is 401/403 (never a silent pass-through)
       - Check `is_dev_bypass_active()` usage — it should only be checked when `cfg!(debug_assertions)` or `AOS_DEV_NO_AUTH=1` is explicitly set
       - If any code path silently passes auth failures in production mode, fix it to return an error

    5. Update `configs/cp.toml` with commented per-tier defaults:
       ```toml
       [rate_limits]
       # health_rpm = (unlimited — health probes not rate-limited)
       # public_rpm = 600
       # internal_rpm = 1000
       # protected_rpm = 300
       ```

    Avoid: Do NOT create a fourth rate limiting implementation — wire the per-tier config into the EXISTING `RateLimiterState` (dashmap-based token bucket). The SQLite-based rate limiter in `security/rate_limiting.rs` is for persistent accounting, not hot-path enforcement. Do NOT modify the middleware chain ordering — rate_limiting already has its position in the global chain.

    1. Create `scripts/contracts/check_security_audit.sh` following existing contract check pattern:
       - Header: `#!/usr/bin/env bash`, `set -euo pipefail`
       - **Auth enforcement check**: Use the existing `check_api_route_tiers.py` and `check_middleware_chain.py` to verify all protected routes have auth middleware. Fail if any protected route lacks auth.
       - **Rate limit coverage check**: Verify all non-health route groups have rate limiting middleware applied. Use grep/ast parsing of the route builder source.
       - **Input validation check**: Scan all Axum handler functions for request body extraction (`Json<T>`, `Query<T>`, `Path<T>`) and verify corresponding types use serde validation or custom validators (grep for `#[validate]` or manual validation in handler body). Report handlers with unvalidated input as warnings.
       - **Security headers check**: Verify CSP, HSTS, X-Frame-Options, Permissions-Policy are set in `middleware_security.rs`. Grep for the header names.
       - **Dev bypass gating check**: Verify `AOS_DEV_NO_AUTH` and `is_dev_bypass_active()` are not compiled into release builds without `cfg(debug_assertions)` guards. Extend existing `check_release_security_assertions.sh` logic.
       - **Dependency vulnerability scan**: Run `cargo audit` and fail on critical/high severity advisories. Integrate with existing `scripts/security_audit.sh`.
       - Output: pass/fail summary with counts. Exit 0 if all pass, exit 1 if any failures.
       - Make executable: `chmod +x`

    2. Create `scripts/contracts/check_secret_exposure.sh`:
       - Scan source code (excluding `target/`, `var/`, `.git/`) for secret patterns:
         - Hardcoded API keys: `api_key\s*=\s*"[^"]{20,}"`, `secret\s*=\s*"[^"]{10,}"`
         - JWT secrets in code: `jwt_secret.*=.*"[^"]+"`
         - Private keys: `-----BEGIN.*PRIVATE KEY-----`
         - Connection strings with credentials: `://[^:]+:[^@]+@`
       - Scan for secrets in error messages:
         - Grep `format!` and `tracing::{error,warn}` calls that reference config fields like `jwt_secret`, `api_key`, `database_url`
         - Grep for `Debug` derives on config structs that contain secret fields (should use custom Debug impls that redact)
       - Scan for secrets in test output:
         - Check that test fixtures don't use real-looking secrets (should use obvious fakes like "test-secret-do-not-use")
       - Output: pass/fail per check category. Exit 0 if clean, exit 1 if any secrets found.
       - Make executable: `chmod +x`

    3. Register both scripts in `scripts/contracts/check_all.sh`:
       - Add `check_security_audit.sh` and `check_secret_exposure.sh` to the list of contract checks that `check_all.sh` runs
       - Maintain the existing pattern (loop through scripts, report pass/fail)

    4. Update `scripts/security_audit.sh` if needed:
       - Ensure it runs `cargo audit` with `--deny warnings` for high/critical
       - Ensure it integrates with the new `check_security_audit.sh` (can be called from it, or standalone)

    Make both scripts follow the exact output format of existing contract checks (look at `check_determinism_contract.sh` for the pattern).

    Avoid: Do NOT scan at runtime — these are CI-time/pre-release checks only. Do NOT implement tracing redaction filters — that's a code change, not a script check. The scripts DETECT issues; the executor fixes detected issues in subsequent tasks if any are found.
