<objective>
Fix training worker binary resolution so it finds the binary deterministically and fails boot with an actionable error when the binary is missing.

Purpose: The current `resolve_training_worker_bin()` falls through all resolution strategies to a bare `"aos-training-worker"` PATH lookup that always fails. This causes the training worker to never spawn on boot.

Output: Config-driven binary path in `PathsConfig`, deterministic resolution function returning `Result`, preflight gate that blocks boot before port bind when binary is missing.
</objective>

  <action>
1. In the training worker supervisor setup block (around line 342-350, inside `if training_execution_mode() == Worker`), add preflight validation BEFORE spawning the supervisor task:
   ```rust
   // Read config to get PathsConfig for binary resolution
   let paths_config = {
       let cfg = state.config.read().map_err(|e| anyhow::anyhow!("Config lock poisoned: {}", e))?;
       cfg.paths.clone()
   };

   // Preflight: resolve and validate training worker binary
   let training_worker_bin = resolve_training_worker_bin(&paths_config)?;
   let bin_path = std::path::Path::new(&training_worker_bin);
   if !bin_path.exists() {
       anyhow::bail!(
           "Training worker binary not found at {}. Build it: cargo build -p adapteros-training-worker",
           training_worker_bin
       );
   }
   if !bin_path.is_file() {
       anyhow::bail!(
           "Training worker path {} exists but is not a file",
           training_worker_bin
       );
   }
   info!(path = %training_worker_bin, "Training worker binary validated at preflight");
   ```
   This runs BEFORE `spawner.spawn_with_details(...)`, so if the binary is missing, boot fails before the supervisor task starts and before the port is bound.

2. Pass the resolved `training_worker_bin` path into the supervisor closure (clone it for the `move ||`). The supervisor should use this pre-resolved path for all spawn attempts, not re-resolve each time.

3. Update `spawn_training_worker()` to take `bin_path: &str` as a parameter and use `Command::new(bin_path)` instead of calling `resolve_training_worker_bin()`.

4. Remove the fallback error handling that signals readiness on binary-not-found (lines ~518-543, the `is_training_worker_fallback_error` branch that writes degraded marker and sends `Ok(())`). Per user decision: no retry on spawn failure for binary-not-found — boot blocks. The fallback code is no longer reachable because preflight catches missing binary before the supervisor starts.

5. Verify that when `resolve_training_worker_bin()` returns `Err`, the error propagates up through `spawn_background_tasks()` and causes boot to fail with the actionable error message containing the path and build instructions.
