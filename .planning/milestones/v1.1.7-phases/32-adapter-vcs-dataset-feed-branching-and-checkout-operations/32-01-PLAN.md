# Phase 32 Plan 01: Ship Checkout-First Contracts and Branch-Aware Dataset Feed

## Objective

Complete the shift from rollback-era semantics to checkout-first adapter version control, while preserving compatibility and making dataset-fed evolution branch-aware.

## Natural-Language Execution Plan

1. Add checkout-compatible route and handler mapping while keeping rollback route valid for existing clients.
2. Promote checkout naming in UI/client contracts and keep rollback alias as compatibility shim.
3. Extend training query/context parsing for optional branch/version feed context from adapter detail/update center flows.
4. Show branch-context cues in training wizard when feed starts from adapter version controls.
5. Verify route + UI alignment with targeted API/UI checks.

## Verification Lens

Success requires both compatibility and clarity: no existing rollback consumers should break, and new operators should experience checkout-first semantics end-to-end.

## Success Criteria

- [x] Checkout-first route/client semantics exist with rollback backward compatibility.
- [x] Branch-aware dataset feed context can be passed from adapter version controls into training flow.
- [x] OpenAPI/client docs and UI text agree on checkout terminology.
- [x] Targeted verification confirms no regression in adapter version mutation flows.
