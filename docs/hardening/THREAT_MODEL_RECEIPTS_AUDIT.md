# Threat Model & Hardening Plan: Receipts and Audit Trails

**Document Version**: 1.0
**Date**: 2026-01-06
**Scope**: Receipt integrity, audit chain verification, and forensic replay for AdapterOS

---

## Threat Model

### Attacker Profiles

#### 1. Malicious Tenant/User

**What they can tamper with**:
- Their own request payloads and parameters
- Submission of falsified receipts for verification
- Replay of old receipts to claim different outputs
- Manipulation of client-side telemetry before submission

**What they cannot tamper with** (assumptions):
- Server-side receipt generation and signing (if key is secure)
- Other tenants' data (tenant isolation assumed)
- Backend attestation reports generated by the worker
- Evidence envelope chain stored server-side

**Detection goals**:
- Receipt signature invalidation on tampered payloads
- Context digest mismatch on modified prompts
- Duplicate receipt detection via trace_id uniqueness

**Prevention goals**:
- Tenant isolation via per-tenant chain scoping
- Signed receipts binding outputs to inputs

---

#### 2. Compromised Worker Process

**What they can tamper with**:
- Per-token decisions (adapter selection, gates)
- Output tokens generated during inference
- Backend/kernel selection reported in receipts
- Seed derivation (if seed context not properly propagated)
- Backend attestation report contents

**What they cannot tamper with** (assumptions):
- Signing key (if stored in Secure Enclave / separate HSM)
- Evidence envelopes already submitted to control plane
- Policy audit decisions already recorded

**Detection goals**:
- `DeterminismViolation::RouterDecisionMismatch` on replayed traces
- `backend_attestation_b3` mismatch if metallib hash differs
- Receipt digest mismatch if run_head_hash is falsified

**Prevention goals**:
- Signing key isolation from worker memory
- Strict mode enforcement blocking fallback seeds
- Backend attestation binding to receipt

---

#### 3. Compromised Control Plane

**What they can tamper with**:
- Receipt storage and retrieval
- Evidence envelope chain (root insertion, deletion)
- Policy audit chain entries
- Telemetry bundle storage and Merkle roots
- Verification endpoint responses

**What they cannot tamper with** (assumptions):
- Externally audited evidence exports (if exported and verified offline)
- Receipts verified by clients using public key
- Diagnostic bundles captured before compromise

**Detection goals**:
- `AuditChainDivergence` when chain is reconstituted from backups
- `AuditExportTamper` when bundle hash differs from expected
- Cross-reference with external audit logs

**Prevention goals**:
- Regular evidence chain exports to cold storage
- Multi-party key custody for signing
- Immutable audit log append-only semantics

---

#### 4. Compromised Disk/Storage

**What they can tamper with**:
- SQLite database files (policy_audit, evidence_envelopes)
- Telemetry bundle files on disk
- Receipt bundle JSON files
- Adapter .aos files and model weights

**What they cannot tamper with** (assumptions):
- In-memory signed receipts (until written)
- Network-transferred receipts (if TLS in use)
- Offsite backups (if replicated before compromise)

**Detection goals**:
- Entry hash recomputation failure on read
- Signature validation failure on receipts
- Chain sequence gaps or `previous_root` mismatches

**Prevention goals**:
- Read-time integrity checks on all policy audit entries
- BLAKE3 hashing of all artifacts at rest
- Encrypted backups with integrity verification

---

#### 5. Honest-but-Buggy Backend (Drift Scenario)

**What can go wrong**:
- Non-deterministic Metal kernel compilation producing different `metallib_hash`
- Floating-point mode variations across hardware
- Unintended fallback to dev/test seed paths
- Thread-local seed context not properly cleared between requests
- CoreML/MLX version differences affecting outputs

**Detection goals**:
- `DeterminismViolation::OutputDrift` on replay comparison
- `backend_attestation_b3` difference flagged in verification
- `ReplayMismatch` in determinism_violation_event

**Prevention goals**:
- Strict mode blocking `NonDeterministic` seed_mode in release
- `assert_thread_local_clean()` enforcement at request boundaries
- Metallib hash pinning in deployment manifests

## Integrity Surfaces Inventory

### 1. Model Hashes

| Aspect | Details |
|--------|---------|
| **Produced** | During model seeding (`adapteros-server/src/model_seeding.rs`) |
| **Stored** | `models` table in SQLite; cached in `ModelCacheIdentityV2` |
| **Validated** | At inference start via manifest hash comparison |
| **Covered in Receipt** | `model_cache_identity_v2_digest_b3` in `InferenceReceiptRef` |
| **Gap** | Tokenizer hash not independently tracked in receipt |

### 2. Adapter Hashes / .aos File Hashes

| Aspect | Details |
|--------|---------|
| **Produced** | During adapter registration (`adapteros-db/src/adapters.rs`) |
| **Stored** | `adapters` table, `adapter_bundle_hash` column |
| **Validated** | At hot-swap and inference via bundle hash verification |
| **Covered in Receipt** | Implicitly via `stack_hash_hex` which hashes adapter ordering |
| **Gap** | Individual adapter bundle hashes not itemized in receipt |

### 3. Router Decision Hash

| Aspect | Details |
|--------|---------|
| **Produced** | Per-token in worker (`hash_decision` in verify_receipt.rs:173-200) |
| **Stored** | In receipt bundle as `decision_hash_hex` per token |
| **Validated** | Recomputed during verification; chained into `run_head_hash` |
| **Covered in Receipt** | Yes - `run_head_hash_hex` is chain of all decision hashes |
| **Gap** | None - well covered |

### 4. Receipts and Signature

| Aspect | Details |
|--------|---------|
| **Produced** | At inference completion (`receipt_digest` from multi-hash) |
| **Stored** | `receipt_bundle.json` in evidence export |
| **Validated** | Ed25519 signature over `receipt_digest` |
| **Current Fields Covered** | context_digest, run_head_hash, output_digest, token accounting (5 fields) |
| **Gaps** | See Receipt Completeness section below |

### 5. Policy Audit Chain

| Aspect | Details |
|--------|---------|
| **Produced** | On each policy hook invocation (`policy_audit.rs`) |
| **Stored** | `policy_audit_decisions` table with `entry_hash`, `previous_hash` |
| **Validated** | `verify_policy_audit_chain()` recomputes and checks linkage |
| **Chain Verification** | Per-tenant, sequence-ordered, hash-linked |
| **Gap** | No automatic periodic verification job; only on-demand |

### 6. Telemetry Bundles and Merkle Roots

| Aspect | Details |
|--------|---------|
| **Produced** | Periodic flush from telemetry buffer (`adapteros-telemetry/src/merkle.rs`) |
| **Stored** | As `BundleMetadataRef` in evidence envelopes |
| **Validated** | Merkle proof verification for individual events |
| **Chain Linking** | Via `EvidenceEnvelope.previous_root` |
| **Gap** | Telemetry verbosity levels may omit events needed for replay |

## Receipt Completeness Recommendations

### Current Receipt Digest Inputs (verify_receipt.rs:442-451)

```rust
B3Hash::hash_multi(&[
    context_digest,           // tenant + stack + prompt
    run_head,                 // chained decision hashes
    output_digest,            // output token sequence
    logical_prompt_tokens,    // token accounting
    prefix_cached_token_count,
    billed_input_tokens,
    logical_output_tokens,
    billed_output_tokens,
])
```

### Recommended Additions

| Field | Rationale | Privacy | Size Impact |
|-------|-----------|---------|-------------|
| `backend_used` | Detect backend substitution (MLX vs CoreML vs Metal) | Safe | 8-16 bytes |
| `backend_attestation_b3` | Bind to specific metallib/compiler state | Safe | 32 bytes |
| `seed_mode_tag` | Distinguish Strict/BestEffort/NonDeterministic | Safe | 1 byte |
| `policy_mask_digest` | Already in context, but explicitly in receipt | Safe | 32 bytes |
| `stack_version` | Version of adapter stack for migration tracking | Safe | 4 bytes |
| `schema_version` | Receipt format version for forward compat | Safe | 1 byte |
| `root_seed_digest` | BLAKE3(request_seed) - NOT raw seed | Safe | 32 bytes |
| `determinism_level` | strict/relaxed/non-deterministic enum | Safe | 1 byte |

### Fields NOT to Include

| Field | Reason |
|-------|--------|
| Raw seed bytes | Allows replay attacks; use digest instead |
| Private key material | Security violation |
| Full policy pack contents | Size explosion; use digest |
| Wall-clock timestamps | Non-deterministic; use logical ticks |

### Versioning Strategy

1. Add `schema_version: u8` to `ReceiptDigests` (default 1 for current)
2. Version 2 adds backend identity fields
3. Version 3 adds seed digest
4. Verification checks schema_version and applies appropriate hash algorithm
5. Maintain backward compatibility: old receipts still verify with schema_version inference

## Verification Workflows

### 1. Local Receipt Verification (CLI)

**Entry Point**: `aosctl verify receipt --bundle <path>`

**Current Flow** (verify_receipt.rs):
1. Load bundle from JSON file
2. Recompute context_digest from tenant + stack + prompts
3. For each token: recompute decision_hash, chain into run_head
4. Compare computed vs expected digests
5. If signature present: Ed25519 verify over receipt_digest
6. Output: `ReceiptVerificationReport` with reason codes

**Failure Handling**:
- `CONTEXT_MISMATCH`: Log tenant/stack details, abort
- `TRACE_TAMPER`: Identify first mismatched token index
- `OUTPUT_MISMATCH`: Flag output token sequence difference
- `SIGNATURE_INVALID`: Distinguish missing vs failed signature

**Observability Events**:
- `receipt_mismatch_event()` on any failure
- Counter: `receipt_mismatch_total`
- Error code: `OBS_RECEIPT_MISMATCH`

### 2. Server-Side Verification on Ingestion

**Proposed Entry Point**: Evidence envelope submission handler

**Flow**:
1. Parse incoming `EvidenceEnvelope`
2. Validate schema_version matches current
3. Verify envelope.root matches recomputed payload digest
4. For inference scope: verify receipt_digest from InferenceReceiptRef
5. Verify signature over canonical envelope bytes
6. Check chain linkage: previous_root matches tail of chain
7. Store envelope with chain_sequence

**Failure Handling**:
- Reject envelope with specific error code
- Emit `AuditChainDivergence` event on linkage failure
- Do NOT store invalid envelopes

**Observability Events**:
- `audit_chain_divergence_event()` on chain break
- Counter: `audit_divergence_total`

### 3. Periodic Audit Chain Verification Job

**Proposed Entry Point**: `aosctl audit verify-chains --all-tenants`

**Flow**:
1. For each tenant:
   a. Load policy audit chain in sequence order
   b. Recompute each entry_hash from fields
   c. Verify entry_hash[n] == recomputed
   d. Verify previous_hash[n] == entry_hash[n-1]
2. For each evidence scope per tenant:
   a. Load evidence envelopes in sequence order
   b. Verify previous_root linkage
   c. Verify signature on each envelope
3. Report divergence at first failure point

**Failure Handling**:
- Log divergence details with chain_sequence
- Emit alert via `audit_chain_divergence_event()`
- Continue checking to find full extent of damage

**Alerting**:
- Prometheus metric: `audit_divergence_total{tenant_id, scope}`
- Alert rule: any divergence triggers PagerDuty

## Risks / Unknowns

### 1. Signing Key Management
- **Risk**: Signing key compromise invalidates all receipt trust
- **Unknown**: Is key stored in Secure Enclave or file?
- **Mitigation**: Document key rotation procedure; consider multi-sig

### 2. Chain Recovery After Divergence
- **Risk**: Detected divergence may leave chain unusable
- **Unknown**: Is there a "heal chain" procedure?
- **Mitigation**: Define recovery runbook with manual approval step

### 3. Performance Impact of Verification
- **Risk**: Per-request signature verification adds latency
- **Unknown**: What is acceptable overhead budget?
- **Mitigation**: Benchmark; consider async verification for non-critical paths

### 4. Telemetry Completeness for Replay
- **Risk**: "stages" verbosity level may omit events needed for full replay
- **Unknown**: Which events are strictly required?
- **Mitigation**: Document minimum verbosity for forensic replay; default to "tokens" in compliance mode

### 5. Cross-Version Receipt Compatibility
- **Risk**: Receipt schema changes may break verification of old receipts
- **Unknown**: How long must old receipts remain verifiable?
- **Mitigation**: Maintain verifier support for all schema versions; never remove fields

### 6. Thread-Local State Leakage
- **Risk**: `assert_thread_local_clean()` only panics in debug builds
- **Unknown**: Can production catch leaks?
- **Mitigation**: Add metrics counter for "cleaned up leaked state" events in release
