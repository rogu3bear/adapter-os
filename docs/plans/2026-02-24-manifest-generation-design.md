# Manifest Generation Design

## Summary

Replace the scattered config surface (cp.toml + .env + CLI flags + env vars) with a single **system manifest** (`configs/manifest.toml`) that describes the full adapterOS instance topology. An interactive wizard in `./start` generates this manifest on first boot.

## Problem

Today, configuring an adapterOS instance requires touching multiple files:

- `configs/cp.toml` for server/db/security/rate-limit tuning
- `.env` for model path, backend, tokenizer, manifests, secrets
- CLI flags (`--skip-worker`, `--worker-backend`, `--quick`) for service topology
- Env vars (`SKIP_WORKER`, `SKIP_SECD`, `SKIP_NODE`) for service selection

New users don't know which levers to pull. The config is split by accident of history, not by design.

## Design

### The Manifest

`configs/manifest.toml` is the single source of truth for an adapterOS instance. It uses TOML format (consistent with the existing ecosystem) and preserves the existing section structure from `cp.toml` for backwards compatibility with the Rust config loader.

New sections added:

| Section | Purpose |
|---------|---------|
| `[instance]` | Instance metadata (name, profile, schema version) |
| `[services]` | Which services to start (backend, worker, secd, node) |
| `[boot]` | Boot behavior (quick, verify_chat, timeouts) |
| `[model]` | Model path, backend type, model manifest reference |

All existing sections (`[server]`, `[db]`, `[security]`, `[paths]`, `[worker.safety]`, `[circuit_breaker]`, `[rate_limits]`, `[metrics]`, `[alerting]`, `[logging]`, `[model_server]`, `[general]`, `[auth]`, `[model.cache]`, `[coreml]`, `[health.adapter]`) are preserved unchanged.

### Manifest Location

- **Path:** `configs/manifest.toml`
- Must be added to `.gitignore` (instance-specific)
- Override: `AOS_CONFIG=path/to/other.toml` bypasses the manifest

### Config Precedence (unchanged conceptually)

```
CLI flags > .env overrides > manifest.toml > profile defaults
```

The `.env` file survives as an optional override layer for secrets and per-session tweaks.

### Profiles

Three built-in profiles provide baseline defaults:

| Profile | Auth | Database | Determinism | Logging |
|---------|------|----------|-------------|---------|
| `development` | dev_login, relaxed JWT | SQLite (local) | besteffort | info, human-readable |
| `production` | enforced JWT, no dev_bypass | SQLite or Postgres | strict | info, JSON format |
| `reference` | dev_login, fixed seeds | SQLite (scoped) | strict, fixed seeds | info+debug, human-readable |

### Model Manifest Relationship

The system manifest *references* a model manifest file (`manifests/*.yaml`), it doesn't inline model/adapter/router/policy config. The model manifests continue to describe the model deployment topology. The system manifest's `[model]` section points to one:

```toml
[model]
path = "var/models/Qwen2.5-7B-Instruct-4bit"
backend = "auto"
manifest = "manifests/qwen7b-4bit-mlx-base-only.yaml"
```

## Wizard Flow

### Trigger

`./start` checks for `configs/manifest.toml` at startup. If missing, the wizard runs automatically. Re-run anytime with `./start init`.

### Steps

1. **Profile selection** — Development / Production / Reference
2. **Model configuration** — Model path (auto-detect from `var/models/`), backend, model manifest
3. **Network** — API port, bind address
4. **Services** — Worker, SecD, Node Agent toggles
5. **Database** — URL (production profile expands this)
6. **Security** — JWT mode, registration (production profile expands this)
7. **Write manifest** — Generate TOML, confirm, add to `.gitignore`

### Implementation

The wizard is a bash function (`generate_manifest()`) in a new sourced library `scripts/lib/manifest-wizard.sh`. No new binary required.

Each profile is a bash function that emits a TOML template string. The wizard substitutes user-provided values into the template. Validation uses the existing `env-loader.sh` validators (port ranges, path existence, etc.).

## Integration

### `./start` Changes

1. **Manifest detection** — At top of `main()`, check `configs/manifest.toml`. If missing, run `generate_manifest()`.

2. **Config resolution** — Change the default config path from `configs/cp.toml` to `configs/manifest.toml` when the manifest exists. `AOS_CONFIG` env var still overrides.

3. **Service topology** — Read `[services]` from manifest via awk/grep to set `SKIP_WORKER`, `SKIP_SECD`, `SKIP_NODE` (bridge to existing service-manager logic). Env vars and CLI flags still override.

4. **Model env bridge** — Read `[model]` section and export `AOS_MODEL_PATH`, `AOS_MODEL_BACKEND`, `AOS_WORKER_MANIFEST` for the service-manager. Only set if not already in env (respects override layer).

5. **Boot config** — Read `[boot]` section for `AOS_QUICK_BOOT`, `AOS_HEALTH_TIMEOUT`, `AOS_READYZ_TIMEOUT`, `AOS_VERIFY_CHAT`.

6. **New command** — `./start init` re-runs the wizard with current manifest values as defaults.

### Rust `EffectiveConfig` Changes

1. Add structs: `InstanceSection`, `ServicesSection`, `BootSection`, `ModelTopologySection`
2. Parse from TOML alongside existing sections
3. `[model].path` / `[model].backend` feed into the places currently reading `AOS_MODEL_PATH` / `AOS_MODEL_BACKEND` from env (at lower precedence than env)
4. Unknown sections already silently ignored — old binaries read the new manifest without breaking

### Backwards Compatibility

- `configs/cp.toml` works if `AOS_CONFIG=configs/cp.toml` is set
- `.env` overrides still apply on top of the manifest
- CLI flags (`--worker-backend`, `--skip-worker`) still override everything
- The server binary doesn't require `[instance]`, `[services]`, or `[boot]` sections to be present

## Files Changed

| File | Change |
|------|--------|
| `start` | Add manifest detection, `init` command, read new sections |
| `scripts/lib/manifest-wizard.sh` | New file: interactive wizard + profile templates |
| `scripts/lib/manifest-reader.sh` | New file: TOML section reader (awk-based, no new deps) |
| `crates/adapteros-config/src/effective.rs` | Add new section structs, parse from TOML |
| `crates/adapteros-config/src/schema.rs` | Add schema entries for new config vars |
| `configs/manifest.toml` | Generated (gitignored) |
| `.gitignore` | Add `configs/manifest.toml` |

## Not In Scope

- Replacing model manifests (`manifests/*.yaml`) — these remain as-is
- Multi-node topology (nodes discover each other at runtime, not via manifest)
- GUI-based config editor
- Config migration tool from cp.toml to manifest (manual or trivial copy)
